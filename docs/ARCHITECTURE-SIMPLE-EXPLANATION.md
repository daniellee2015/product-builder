# Product Builder 架构 - 大白话版

## 核心思想：两个独立的工作流

想象你是一个项目经理，手下有一个开发团队。

### 工作流1：你（项目经理）的工作

**你要做什么：**
1. 听用户说需求
2. 把大需求拆成小任务
3. 决定先做哪个，后做哪个
4. 盯着进度，看有没有卡住的地方
5. 跟用户汇报进展

**这就是"调度工作流"**

**谁来做：**
- 选项A：用OpenClaw这个工具（就像雇一个外部项目经理）
- 选项B：Product Builder自己做（就像公司内部的项目经理）

**重点：只能选一个！**

### 工作流2：开发团队的工作

**开发团队要做什么：**
1. 收集需求细节（Phase 0）
2. 做计划（Phase 1）
3. 写代码（Phase 2）
4. 测试和审查（Phase 3）
5. 整理文档（Phase 4）

**这就是"开发工作流"**

**谁来做：**
- CCB和CCA这些工具
- 可以配置：Phase 0用Claude，Phase 1用Gemini，Phase 2用Codex

**开发团队还要判断：**
- 这两个任务能不能同时做？
- 需不需要分开工作区？
- 会不会冲突？

## 关系

```
你（项目经理）
    ↓ 分配任务
开发团队
    ↓ 干活
完成任务
```

## Codex哪里理解错了？

Codex以为：
- OpenClaw和Product Builder会**同时**当项目经理
- 两个项目经理会打架

**实际上：**
- 只有一个项目经理
- 要么是OpenClaw，要么是Product Builder自己
- 不会同时存在

## 为什么要分两个工作流？

**项目经理关心的：**
- 整个项目怎么样了？
- 哪个任务重要？
- 有没有延期？

**开发团队关心的：**
- 这个代码怎么写？
- 会不会冲突？
- 能不能并行？

**两个人关心的事情不一样，所以是两个独立的流程！**

## Product Builder要配置什么？

### 配置1：选项目经理

```json
{
  "用谁当项目经理": "OpenClaw",  // 或者 "自己"

  "如果用OpenClaw": {
    "地址": "http://localhost:8080"
  },

  "如果自己做": {
    "用哪个AI": "Claude"
  }
}
```

### 配置2：开发团队用哪些AI

```json
{
  "Phase 0（收集需求）": "用Claude",
  "Phase 1（做计划）": "用Gemini",
  "Phase 2（写代码）": "用Codex",
  "Phase 3（测试）": "用Claude",
  "Phase 4（整理）": "用Claude"
}
```

### 配置3：技术规则

```json
{
  "自动判断能不能并行": "开启",
  "自动隔离工作区": "开启",
  "自动检测冲突": "开启"
}
```

## 实际例子

### 用户说："我要做一个认证系统"

**第1步：项目经理（OpenClaw或Product Builder）分析**
```
项目经理想：这个需求太大了，拆成3个小任务：
- 任务1：设计数据库
- 任务2：做OAuth登录
- 任务3：做JWT token

任务1必须先做，任务2和任务3可以同时做
```

**第2步：开发团队干活**
```
任务1：
- Phase 0: 收集需求（用Claude）
- Phase 1: 设计方案（用Gemini）
- Phase 2: 写代码（用Codex）
- Phase 3: 测试（用Claude）
- Phase 4: 整理文档（用Claude）

任务1完成后，任务2和任务3同时开始：

任务2（OAuth）：
- 判断：跟任务3独立吗？是的
- 判断：需要隔离工作区吗？是的
- 创建独立的工作区
- 执行Phase 0-4

任务3（JWT）：
- 判断：跟任务2独立吗？是的
- 判断：需要隔离工作区吗？是的
- 创建独立的工作区
- 执行Phase 0-4
```

**第3步：项目经理跟踪进度**
```
项目经理看：
- 任务1：完成 ✓
- 任务2：进行中 50%
- 任务3：进行中 30%

发现任务3比较慢，提醒用户
```

## 总结

### 最简单的理解

1. **两个角色**
   - 项目经理（PM/Traffic）
   - 开发团队（CCB/CCA）

2. **两个工作流**
   - 项目经理的工作流（管理）
   - 开发团队的工作流（干活）

3. **各干各的**
   - 项目经理：拆任务、定优先级、盯进度
   - 开发团队：写代码、判断技术问题

4. **不混在一起**
   - 项目经理不管怎么写代码
   - 开发团队不管整体优先级

### Codex的误解

Codex把项目经理和开发团队的工作混在一起理解了，所以觉得会冲突。

实际上它们是两个完全独立的流程，不会冲突。

### 关键点

- OpenClaw **或** Product Builder自己，二选一当项目经理
- 项目经理的工作流和开发工作流是**两个独立的流程**
- 不能混在一起

这样说清楚了吗？
